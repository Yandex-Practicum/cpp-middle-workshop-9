### Предложение для самостоятельной работы ###

После воркшопа вы можете попробовать расширить функционал системы, выполнив одно или несколько заданий. Эти задания опциональны, но помогут вам глубже освоить модель senders/receivers.

Данное описание задач доступно в репозитории проекта в ветке `main` в файле `homework/README.md`.

#### Задание 1: Обработка отмены операции через `set_stopped` ####

**Цель:** Научиться корректно обрабатывать отмену операций с использованием `set_stopped`.

**Задача:** Модифицируйте `read_and_print` так, чтобы при получении сигнала остановки от планировщика (например, при вызове `pool.request_stop()`) сендер корректно завершался через `set_stopped`, а не просто прерывал цикл.

**Что нужно сделать:**

1. Используйте `ex::get_stop_token(ctx)` внутри кастомного сендера для получения токена отмены
2. Проверяйте `stop_token.stop_requested()` перед началом операции
3. При запросе остановки вызывайте `ex::set_stopped(std::move(ctx.receiver))` вместо `ex::set_value`
4. Добавьте обработку `set_stopped` в цепочке через `ex::let_stopped` (если доступно) или через кастомный ресивер


#### Задание 2: Написание собственного ресивера ####

**Цель:** Понять внутреннее устройство ресиверов и научиться создавать их самостоятельно.

**Задача:** Реализуйте собственный ресивер для обработки результатов работы сендеров без использования `ex::sync_wait`. Ресивер должен логировать все три типа сигналов завершения.

**Что нужно сделать:**

1. Создайте структуру ресивера с тремя методами:

   ```cpp
   struct MyReceiver {
       void set_value(OrderBook book) { /* логирование значения */ }
       void set_error(std::exception_ptr eptr) { /* логирование ошибки */ }
       void set_stopped() noexcept { /* логирование отмены */ }
   };
   ```

2. Реализуйте операцию соединения сендера с ресивером через `ex::connect(sender, receiver)`
3. Запустите операцию через `ex::start(operation_state)`
4. Используйте ваш ресивер для обработки данных от одного из серверов

**Ожидаемый результат:**

```
[MyReceiver] set_value called: ServerA - Bid: $100.00, Ask: $100.50
[MyReceiver] set_value called: ServerB - Bid: $101.50, Ask: $102.00
[MyReceiver] set_stopped called: operation cancelled
```

#### Задание 3: Сравнение алгоритмов переключения контекста ####

**Цель:** Понять различия между `ex::on`, `ex::continues_on` и `ex::schedule`.

**Задача:** Создайте три версии конвейера обработки данных, каждая из которых использует один из алгоритмов переключения контекста. Замерьте время выполнения и выведите ID потоков на каждом этапе.

**Что нужно сделать:**

1. **Версия с `ex::on`:**
   ```cpp
   auto pipeline_on = ex::on(scheduler, ex::just() | ex::then([]{ /* работа */ }));
   ```
2. **Версия с `ex::continues_on`:**
   ```cpp
   auto pipeline_continues = ex::just() | ex::continues_on(scheduler) | ex::then([]{ /* работа */ });
   ```
3. **Версия с `ex::schedule`:**
   ```cpp
   auto pipeline_schedule = ex::schedule(scheduler) | ex::then([]{ /* работа */ });
   ```
4. Для каждой версии выведите `std::this_thread::get_id()` до, во время и после выполнения
5. Сравните поведение: где выполняется каждый этап, возвращается ли управление в исходный контекст
